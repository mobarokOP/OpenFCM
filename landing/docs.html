Introduction

OneFCM is a self-hosted push-notification platform for Android. You drop one Kotlin SDK into your app, connect your Firebase project with a single file in the dashboard, and send targeted notifications from the dashboard or the REST API — by user, tag, segment, or topic.

Architecture in three sentences. The Android SDK registers each device with the OneFCM backend and fetches your app's Firebase client config at runtime — no google-services.json in your project. When you send, the backend resolves the audience, fans devices out in batches onto a queue, and delivers through Firebase Cloud Messaging HTTP v1 using your own service account, with exponential-backoff retries and dead-token cleanup. Every attempt is written to delivery logs, and received/opened events flow back for analytics (sent, delivered, opened, CTR).

💡
New here? The fastest path is: create an app → connect Firebase → install the SDK → initialize. That's a working push setup in under ten minutes.

Create an application

Every Android app you send to is an application in OneFCM, identified by a public App ID.

  1. Sign in at dashboard.onefcm.com.
  2. Go to Applications → New application.
  3. Enter a name and your app's Android package name (e.g. com.example.reader) — it must match the applicationId in your Gradle config.
  4. Save, then copy the App ID from the app's overview page. You'll pass it to OpenFCM.init(...).
🔑
The App ID is a public identifier — it's safe to ship inside your APK. Server-side secrets (API keys, the Firebase service account) never leave the backend.

Connect Firebase (one file)

OneFCM delivers through your own Firebase project. Connecting it takes one JSON file:

  1. Open the Firebase Console → your project → Project settings → Service accounts.
  2. Click Generate new private key and download the JSON file.
  3. In the OneFCM dashboard, open your app's Settings and upload that JSON.
  4. Wait for the Firebase status to show Connected. Done.
✨
Everything else is derived automatically. The backend uses the service account to pull the client config (project id, sender id, API key, app id) via the Firebase Management API — and if your Firebase project has no Android app yet, one is auto-registered with your package name. Your client app needs no google-services.json and no Firebase setup of its own.

Each application sends through its own Firebase project, so notifications never leak between apps.

Install the SDK

The SDK is published on JitPack and supports minSdk 24+. Add the JitPack repository, then the dependency.

settings.gradle.kts
dependencyResolutionManagement {
    repositories {
        google()
        mavenCentral()
        maven { url = uri("https://jitpack.io") }   // add this
    }
}
app/build.gradle.kts
dependencies {
    implementation("com.github.mobarokOP:OneFCM:1.0.0")
}
📦
Firebase Messaging is pulled in transitively — you don't add any FCM dependency or Google Services Gradle plugin yourself.

Initialize

Initialize once in your Application class with the App ID you copied in Create an application.

MyApp.kt
import android.app.Application
import com.openfcm.sdk.OpenFCM

class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        OpenFCM.init(this, appId = "YOUR_APP_ID")
    }
}

One line is all it takes — the SDK talks to the OneFCM cloud by default. An optional config block customizes channels, icons and logging, or points the SDK at a self-hosted server:

MyApp.kt · optional config
OpenFCM.init(this, appId = "YOUR_APP_ID") {
    baseUrl = "https://push.example.com" // self-hosted only
    defaultChannelId = "general"
    defaultChannelName = "General"
    enableDebugLogging = BuildConfig.DEBUG
}

Register the class in your manifest:

AndroidManifest.xml
<application
    android:name=".MyApp"
    ... >
📱
That's it — the device auto-registers on first launch: the SDK fetches your Firebase config from the server, initializes Firebase, and syncs the FCM token. The SDK's messaging service is merged in via the library manifest, so there's nothing else to declare. Verify under dashboard → your app → Devices.

Notification permission

On Android 13+ the runtime permission is requested automatically on your first Activity — nothing to do by default. Prefer to control the moment yourself? Opt out and prompt manually:

Permission.kt
// 1 · Opt out of the automatic prompt in init { }
OpenFCM.init(this, appId = "YOUR_APP_ID") {
    promptForPermissionOnInit = false
}

// 2 · Ask at the right moment in your UX (Android 13+)
OpenFCM.requestNotificationPermission(activity)

// Check the current state at any time
val enabled = OpenFCM.areNotificationsEnabled()

Below Android 13 no runtime permission exists; notifications work as soon as the device registers.

Users, tags & topics

Three lightweight primitives drive all targeting. Call them anywhere — every call is main-thread-safe, queues before init completes, and retries offline via WorkManager.

Targeting.kt
// Users — link this device to your own user id (after sign-in)
OpenFCM.login("user_12345")
OpenFCM.logout()                       // on sign-out

// Tags — key/value attributes on the device
OpenFCM.addTag("premium", "true")
OpenFCM.addTags(mapOf("language" to "bn", "city" to "Rangpur"))
OpenFCM.removeTag("premium")

// Topics — broadcast channels the device opts into
OpenFCM.subscribeTopic("announcements")
OpenFCM.unsubscribeTopic("announcements")
PrimitiveWhat it enables
login(externalId)Target exact people with the user_ids audience — across all their devices.
addTag / addTagsFilter by attributes with the tags audience (e.g. premium eq "true") and build dynamic segments in the dashboard.
subscribeTopicBroadcast to everyone subscribed via the topic audience — great for opt-in channels like “announcements”.

Deep links & tap handling

Set the deep_link field when sending (e.g. myapp://book/25). On tap, the SDK first tries to resolve it against an intent-filter inside your app; if nothing matches and it's an http(s) URL, it opens the browser instead.

For full control — including custom data key routing — register a tap handler:

TapHandler.kt
OpenFCM.setNotificationOpenHandler { payload ->
    // payload.notificationId · payload.deepLink · payload.data[...]
    val bookId = payload.data["book_id"]
    when {
        bookId != null -> startActivity(
            Intent(this, BookActivity::class.java)
                .putExtra("book_id", bookId)
        )
        else -> payload.deepLink?.let { openDeepLink(it) }
    }
}

The matching send payload would include "deep_link": "myapp://book/25" and "data": {"book_id": "25"} — see Send a notification. Open events are tracked automatically and appear as opened / CTR in analytics.

Troubleshooting

SymptomCheckFix
Device not registering App ID in OpenFCM.init (and baseUrl, if self-hosting); Logcat filtered by OpenFCM (with enableDebugLogging = true). Use the exact App ID from the dashboard; if self-hosting, the baseUrl must be an HTTPS URL reachable from the device. The Logcat lines name the failing request.
Notification not arriving Dashboard → app Settings: Firebase status Connected? Notification permission granted? Emulator image includes Google Play? Re-upload the service-account JSON if not connected; prompt for permission; use a Google Play emulator image — plain AOSP images have no FCM.
Image not showing Is image_url a direct, publicly reachable image URL? Link the image file itself over HTTPS (e.g. …/cover.jpg), not an HTML page or a login-protected URL.
⚠️
OEM battery savers (Xiaomi/MIUI, Huawei, Oppo, Vivo, some Samsung modes) aggressively kill background processes and can delay or drop pushes on those devices. Ask affected users to exempt your app from battery optimization — this is an OS-level behaviour no push provider can bypass.

REST API — Authentication

Send server-to-server with an API key. In the dashboard open your app → API Keys → New key. The full key (op_live_…) is shown once — store it in a secret manager. Keys belong to one application and can be revoked any time.

Base URL: https://api.onefcm.com/v1 · Content type: application/json · Auth header on every request:

auth.sh
curl https://api.onefcm.com/v1/notifications/NOTIFICATION_ID \
  -H "Authorization: Bearer op_live_YOUR_API_KEY"

All timestamps are ISO-8601 UTC; IDs are UUID strings. Errors use one shape — see Error codes.

Send a notification

One endpoint sends everything: POST /v1/notifications. Only app_id, title, body, and audience are required — the rest is optional.

send.sh
curl -X POST https://api.onefcm.com/v1/notifications \
  -H "Authorization: Bearer op_live_YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "app_id": "YOUR_APP_ID",
    "title": "New chapter published! 📖",
    "body": "Tap to read the latest update.",
    "image_url": "https://cdn.example.com/covers/25.jpg",
    "deep_link": "myapp://book/25",
    "data": { "book_id": "25" },
    "priority": "high",
    "ttl": 2419200,
    "audience": {
      "type": "tags",
      "value": [{ "field": "premium", "op": "eq", "value": "true" }]
    }
  }'

The response confirms queueing and the resolved audience size:

201 · response.json
{
  "data": {
    "id": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
    "status": "queued",
    "estimated_recipients": 4210
  }
}

Optional fields: image_url (big picture), deep_link, data (custom key/values delivered to your tap handler), priority (high | normal), ttl (seconds, default 2419200 = 28 days), collapse_key, channel_id, and schedule — see Scheduling.

Audiences

The audience object decides who receives the notification. Six types:

TypeTargetsExample audience
allEvery registered device of the app{"type":"all"}
user_idsDevices linked via OpenFCM.login(){"type":"user_ids","value":["user_12345","user_67890"]}
device_idsSpecific device UUIDs{"type":"device_ids","value":["7c9e6679-7425-40de-944b-e07fc1f90ae7"]}
tagsDevices matching every tag filter (AND){"type":"tags","value":[{"field":"premium","op":"eq","value":"true"}]}
segmentA saved segment built in the dashboard{"type":"segment","value":"SEGMENT_ID"}
topicDevices subscribed via subscribeTopic(){"type":"topic","value":"announcements"}

Tag filter operators: eq, neq, exists, contains — the same operators power dynamic segments.

Scheduling

Add a schedule object to any send payload to deliver later instead of now. The response comes back with "status": "scheduled".

schedule.json
// One-time delivery
"schedule": { "send_at": "2026-08-01T09:00:00Z" }

// Recurring — daily | weekly | monthly, anchored to send_at
"schedule": {
  "send_at": "2026-08-01T09:00:00Z",
  "timezone": "Asia/Dhaka",
  "recurring": "weekly"
}

send_at is ISO-8601 UTC; timezone optionally localizes the recurring time. The scheduler promotes due sends every minute. Cancel a scheduled notification any time from the dashboard (Notifications → Scheduled → Cancel).

Check status

Poll any notification by id to follow delivery and engagement:

status.sh
curl https://api.onefcm.com/v1/notifications/7c9e6679-7425-40de-944b-e07fc1f90ae7 \
  -H "Authorization: Bearer op_live_YOUR_API_KEY"
200 · response.json
{
  "data": {
    "id": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
    "title": "New chapter published! 📖",
    "status": "completed",
    "stats": {
      "sent": 4210,
      "delivered": 4102,
      "failed": 108,
      "opened": 1873,
      "ctr": 45.7
    }
  }
}

Per-device outcomes (FCM message id, error, retry count) live in the dashboard's delivery logs for each notification.

MemberDescription

login(externalId)Associate this device with your user id.
logout()Clear the user association.
addTag(key, value)Set a single targeting tag.
addTags(map)Set multiple tags at once.
removeTag(key)Remove a single tag.

unsubscribeTopic(name)Unsubscribe from a topic.

OptionPurpose

🧵
All calls are safe on the main thread and before init completes — they queue internally and never block the UI. Failed network calls are retried offline via WorkManager.

Error codes

Every error uses one shape:

error.json
{
  "error": {
    "code": "validation_error",
    "message": "Human readable",
    "details": { "field": ["msg"] }
  }
}
CodeHTTPMeaningFix
validation_error422The request body failed validation.Inspect error.details — it lists the offending fields and messages.
unauthenticated401Missing or malformed Authorization header.Send Authorization: Bearer op_live_… on every request.
invalid_api_key401The key is unknown or was revoked.Create a new key under App → API Keys and update your secret store.
not_found404The resource doesn't exist or belongs to another app.Check the id and that the key's application matches app_id.
fcm_not_configured404The app has no connected Firebase project.Upload the service-account JSON — see Connect Firebase.
in_flight422The action conflicts with a send already in progress (e.g. cancelling a completed notification).Wait for the current state to settle, then retry or check status first.
